## 1. 项目概述

### 1.1 项目背景

本系统为 **ISO 11820 建筑材料不燃性试验** 的仿真软件。在真实的建材防火检测中，需要将样品放入加热炉升温至 750°C，记录 60 分钟的温度变化，根据温升和失重率判定材料是否不燃。由于真实设备昂贵，本系统通过软件仿真的方式完整模拟这一试验流程。

### 1.2 项目目标

- 实现完整的试验流程仿真（5 个温度通道）
- 实现标准的试验状态机（5 个状态）
- 实时显示温度数据和曲线
- 自动生成 CSV / Excel / PDF 报告
- 支持试验记录查询和校准管理

### 1.3 技术栈

| 技术 | 用途 |
|------|------|
| C# / .NET 8 | 开发语言与框架 |
| Windows Forms | 桌面 UI |
| SQLite | 本地数据库 |
| OxyPlot | 实时温度曲线 |
| EPPlus | Excel 报告生成 |
| PDFsharp-MigraDoc | PDF 报告生成 |
| Serilog | 结构化日志 |
| MathNet.Numerics | 温漂线性回归计算 |

## 3. 技术方案实现

### 3.1 温度仿真引擎

**实现方案**：分阶段物理仿真模型

```
升温阶段 (Preparing):
  线性升温 + 高斯噪声模拟真实传感器波动
  TF1/2 同步上升（独立噪声）
  TS/TC 低比例跟随

稳定阶段 (Ready):
  钳位到 750°C + 微小波动
  稳定计数器判定（连续 4 tick 温度在范围内）

记录阶段 (Recording):
  炉温保持 750°C
  样品温度指数逼近：TS→TF1×0.95，TC→TF1×0.85
  模拟热传导延迟效应
```

**关键参数**：

| 参数 | 值 | 说明 |
|------|:---:|------|
| 目标温度 | 750°C | ISO 标准要求 |
| 升温速率 | 40°C/s | 演示加速模式 |
| 温度波动 | ±0.5°C | 噪声幅度 |
| 稳定阈值 | 3°C | 745~755°C 范围 |
| 采样间隔 | 800ms | 仿真时钟 |

### 3.2 状态机实现

**5 状态模型**：

```
Idle → Preparing → Ready → Recording → Complete
         ↑                        │
         └────────────────────────┘
```

**安全保护机制**：
- 未保存试验阻止新建（`flag != "10000000"`）
- 按钮状态自动跟随状态机
- 温度回退自动回退到 Preparing
- 跨线程操作使用 `lock` 同步

### 3.3 数据库设计

**6 表结构**：

| 表名 | 记录数 | 用途 |
|------|:---:|------|
| operators | 2 | 用户账号 |
| apparatus | 1 | 设备信息 |
| productmaster | 按试验 | 样品信息 |
| testmaster | 按试验 | 试验记录（核心） |
| sensors | 17 | 传感器配置 |
| CalibrationRecords | 按校准 | 校准历史 |

**设计特点**：
- SQLite 文件数据库，零配置
- 温度时序数据不入库，存 CSV 文件
- 报告编号自动生成
- 支持联合主键（productid + testid）

### 3.4 报告生成

**三种格式**：

```
CSV:
  逐秒温度数据，标准逗号分隔格式
  存储路径：TestData/{productid}/{testid}/sensor_data.csv

Excel (EPPlus):
  Sheet1：试验信息表（含判定结论）
  Sheet2：温度数据表
  Sheet3：温度曲线图（嵌入 OxyPlot 图表）

PDF (PDFsharp-MigraDoc):
  报告封面 → 试验概要 → 曲线图 → 判定结论
```

### 3.5 技术创新点

| 创新点 | 说明 |
|--------|------|
| **分阶段仿真模型** | 升温/稳定/记录三个阶段使用不同算法，模拟真实热力学过程 |
| **指数逼近算法** | 记录阶段样品温度使用指数逼近模型，模拟热传导延迟 |
| **配置驱动架构** | 所有仿真参数可配置，无需修改代码即可调整行为 |
| **仿真/硬件双模式** | 同一套代码可切换仿真或真实硬件模式（架构预留） |
| **AI 辅助开发** | 使用 CodeBuddy AI 工具辅助设计、编码、文档生成 |

## 8. 经验总结

### 8.1 技术收获

1. **分层架构的价值**：清晰的分层设计让每个模块职责明确，修改 UI 不影响核心逻辑，修改数据库不影响服务层。

2. **事件驱动模式**：使用 `DataBroadcast` 事件将数据从后台线程传到 UI，避免了紧耦合，也解决了跨线程问题。

3. **配置驱动的灵活性**：将升温速率、目标温度等参数放入 `appsettings.json`，无需重新编译即可调整仿真行为。

4. **SQLite 的适用场景**：对于单机桌面应用，SQLite 是完美的选择——零配置、文件即数据库、足够轻量。

### 8.2 过程反思

1. **AI 辅助开发效率**：使用 CodeBuddy 可以快速生成代码框架、编写文档、调试问题，但需要人工验证生成代码的正确性。

2. **文档先行的重要性**：先写好需求分析、概览设计、详细设计，编码时思路更清晰，减少返工。

3. **状态机设计的严谨性**：状态转换条件必须完整覆盖所有场景，遗漏任何一个转换都可能导致系统行为异常。

4. **跨线程问题的教训**：WinForms 中所有 UI 操作必须在 UI 线程，使用 `Invoke` 是必须遵守的规则。

### 8.3 关键决策回顾

| 决策 | 选择 | 替代方案 | 评价 |
|------|------|---------|:---:|
| UI 框架 | WinForms | WPF/Avalonia | ✅ 学习成本低，适合教学 |
| 数据库 | SQLite | SQL Server/MySQL | ✅ 零配置，完美适配 |
| ORM | 原生 SQL | EF Core | ✅ SQL 透明，易调试 |
| 图表库 | OxyPlot | LiveCharts/ScottPlot | ✅ 成熟稳定 |
| 开发方式 | AI 辅助 | 纯手写 | ✅ 效率提升明显 |
